1) Problem domain + motivation (why necessary?)
- Modern CI/CD security creates a triage bottleneck, not just a detection problem: many findings, duplicate alerts, inconsistent severities, low actionability.
- Existing prioritization (e.g., CVSS-only or tool-native severity) is often context-blind to your codebase’s actual exposure and development risk.
- Security teams have limited remediation capacity, so poor prioritization leads to:
  - delayed fixes for truly risky issues,
  - wasted effort on low-impact findings,
  - alert fatigue and reduced trust in pipeline outputs.
- Your thesis is necessary because it targets the practical gap between “finding vulnerabilities” and “deciding what to fix first in this repository, now.”
Use this logic:  
More tools -> more findings -> higher triage cost -> lower decision quality -> need context-aware prioritization.
2) Methodological approach (what exactly will you build?)
Frame it as Design Science / artifact-based research:
- You conceptualize and implement a reachability-aware prioritization artifact for CI/CD.
- Artifact components:
  - static signal extraction (CodeQL fan-in from entry points),
  - repository evolution signals (git churn/activity),
  - LLM semantic analysis from repository context,
  - orchestration layer that merges normalized JSON outputs into ranked priorities.
- Development process:
  - define ranking model and schema,
  - implement pipeline stages,
  - run iterative refinement on real project(s),
  - compare against baseline prioritization approaches.
- Outcome is not just software, but a method: reproducible workflow + scoring/ranking logic + evaluation protocol.
3) Data, experiments, analysis, and link to RQ
Since there’s no perfect “correctness,” use multi-evidence evaluation.
Data you can collect:
- vulnerability findings from multiple scanners (type, severity, location, tool source),
- reachability metrics (fan-in, entry-point distance/proxy),
- git metrics (churn, recency, ownership/activity),
- LLM ranking outputs + rationale/confidence,
- human/expert triage labels (if available) or proxy labels (KEV/EPSS/CVSS thresholds, exploit references),
- operational data (triage time, top-k review outcomes).
Experiments:
- baseline comparison:
  - tool severity only,
  - CVSS-only,
  - simple heuristic weighting,
  - your combined approach.
- ablations:
  - remove each signal (reachability / churn / LLM) to measure marginal value.
- robustness:
  - repeated runs, prompt/model variations.
- optional operational simulation:
  - “given fixed analyst capacity, how many high-risk findings appear in top-k?”
Analysis:
- ranking metrics: Precision@k, Recall@k, NDCG, top-k hit rate.
- agreement/correlation: Spearman/Kendall with analyst ranking or proxy ranking.
- efficiency: median triage time, issues reviewed before first high-impact finding.
- qualitative: short thematic analysis of where LLM helped/hurt prioritization.
How this answers RQ:
- If your method consistently improves top-k relevance and triage efficiency vs baselines, you show LLM+reachability adds practical prioritization value.
4) Expected results / contributions
State contributions as 3-4 concrete deliverables:
- A reproducible CI/CD prioritization pipeline integrating static reachability, development activity, and LLM semantic signals.
- An evaluation framework for vulnerability prioritization under weak/no ground truth (proxy + expert + operational metrics).
- Empirical evidence on whether combined signals improve triage focus compared to severity-only baselines.
- Practical guidance for adaptive scanning depth (deeper analysis for high-priority regions, lighter checks for lower-priority regions).
You can describe expected impact as:
- better allocation of security effort,
- reduced alert overload,
- more transparent and context-aware remediation decisions.
